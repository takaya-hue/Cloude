# Product Dev Bot (PDB) システム説明書

## 1. システム概要

### 1.1 目的

Product Dev Bot（PDB）は、商品開発プロジェクトの「立ち上げ」「担当者管理」「ワークフロー定義」を Google Chat 上で効率化するボットです。Google Workspace Add-on として動作し、Google Chat から対話的にプロジェクトを登録し、Google Sheets / Google Drive と連携して情報を一元管理します。

### 1.2 システム構成

| コンポーネント | 説明 |
|---|---|
| **Google Chat Bot** | ユーザーとの対話インターフェース（Card V2 UI） |
| **Google Apps Script (GAS)** | ボットのバックエンド処理 |
| **Google Sheets** | データベース（担当者マスタ、タスクテンプレート等） |
| **Google Drive** | プロジェクト管理シートの保存先 |
| **People API** | Google Workspace ディレクトリからのユーザー情報取得 |

### 1.3 デプロイ形態

- **Google Workspace Add-on** としてデプロイ
- GAS プロジェクトを `clasp` で管理
- 全 API 呼び出しは REST API 経由（`UrlFetchApp.fetch()`）

## 2. 初期セットアップ

### 2.1 前提条件

- Google Workspace アカウント
- GCP プロジェクト（Sheets API、Drive API、People API が有効であること）
- `clasp` CLI がインストール済みであること

### 2.2 GCP プロジェクトの設定

以下の API を GCP コンソールで有効にしてください：

1. **Google Sheets API** - スプレッドシートの読み書き
2. **Google Drive API** - ファイルの作成・管理
3. **People API** - Workspace ディレクトリからのユーザー取得

### 2.3 OAuth スコープ

本システムは以下の OAuth スコープを使用します（`appsscript.json` に定義済み）：

| スコープ | 用途 |
|---|---|
| `https://www.googleapis.com/auth/spreadsheets` | スプレッドシートの読み書き |
| `https://www.googleapis.com/auth/drive` | Drive 上のファイル作成・管理 |
| `https://www.googleapis.com/auth/script.external_request` | REST API 呼び出し（UrlFetchApp） |
| `https://www.googleapis.com/auth/directory.readonly` | Workspace ディレクトリの読み取り |

### 2.4 初期セットアップ手順

#### ステップ 1: スクリプトプロパティの設定

GAS エディタの **プロジェクトの設定 > スクリプトプロパティ** で以下を設定します：

| プロパティ名 | 説明 |
|---|---|
| `SPREADSHEET_ID` | データベース用スプレッドシートの ID |

#### ステップ 2: OAuth 認可

GAS エディタで `authorizeScopes` 関数を実行し、すべてのスコープの認可を行います。

```
実行手順: GAS エディタ > 関数選択: authorizeScopes > 実行
```

認可画面が表示されたら、すべてのアクセス許可を承認してください。

#### ステップ 3: データベースの初期化

GAS エディタで `setupDatabase` 関数を実行します。以下のシートが自動作成されます：

| シート名 | 用途 |
|---|---|
| タスクテンプレート | タスクのテンプレート定義 |
| プロジェクト・メンバー対応表 | プロジェクトとメンバーの紐付け |
| 実行タスクリスト | 実行中タスクの管理 |
| 担当者マスタ | 担当者（メンバー）情報の管理 |

#### ステップ 4: ワークフロー定義スプレッドシートの作成

GAS エディタで `createWorkflowSheet` 関数を実行します。

- 指定フォルダ内に「ワークフロー定義」スプレッドシートが作成されます
- スクリプトプロパティ `WORKFLOW_SPREADSHEET_ID` が自動設定されます
- 初期ワークフロー（新商品・リニューアル）が書き込まれます
- 「担当者・ロール定義」シートが作成されます
- People API から取得したディレクトリユーザー一覧が隠しシートに保存されます

#### ステップ 5: デプロイ

```bash
clasp push --force
clasp deploy -i <デプロイメントID>
```

## 3. 機能説明

### 3.1 Google Chat コマンド一覧

Bot にメンション付きでメッセージを送ると、キーワードに応じた機能が呼び出されます。

| キーワード | 機能 |
|---|---|
| `登録` / `立ち上げ` / `新規` / `プロジェクト` | プロジェクト立ち上げフローを開始 |
| `担当者登録` / `メンバー登録` | 担当者登録フォームを表示 |
| 上記以外 | 利用可能なコマンドのヘルプを表示 |

---

### 3.2 プロジェクト立ち上げ機能

商品開発プロジェクトを登録し、管理用スプレッドシートを自動作成する機能です。

#### フロー

```
1. ユーザーが「登録」等のキーワードを送信
    ↓
2. 種別選択カードが表示される
    - 新商品
    - リニューアル・軽微な変更
    - PB（プライベートブランド）
    ↓
3-a. PBの場合 → 配合種別選択カードが表示
    - 新規配合 → PB_NEW_FORMULA
    - 既存配合 → PB_EXISTING_FORMULA
    ↓
3-b. その他 → 直接プロジェクト情報入力へ
    ↓
4. プロジェクト情報入力フォームが表示される
    - 商品名（必須）
    - 規格
    - 発売予定日
    ↓
5. 「登録する」ボタンを押すと：
    - 指定フォルダに管理スプレッドシートが作成される
    - ファイル名: 「種別_商品名_規格_YYMMDD」
    - 確認メッセージが表示される
```

#### プロジェクト種別一覧

| コード | 表示名 | 説明 |
|---|---|---|
| `NEW_PRODUCT` | 新商品 | 新規商品の開発 |
| `RENEWAL` | リニューアル・軽微な変更 | 既存商品の変更 |
| `PB_NEW_FORMULA` | PB - 新規配合 | PB商品（新規配合） |
| `PB_EXISTING_FORMULA` | PB - 既存配合 | PB商品（既存配合流用） |

#### 作成されるスプレッドシート

プロジェクト登録時に作成されるスプレッドシートには「プロジェクト概要」シートが含まれ、以下の情報が記載されます：

| 項目 | 内容例 |
|---|---|
| 種別 | 新商品 |
| 商品名 | ウルトラ花ちゃん培養土 |
| 規格 | 30L |
| 発売予定日 | 2026/06/01 |
| 登録日 | 2026/03/04 15:30 |
| ステータス | 立ち上げ |

---

### 3.3 担当者登録機能

プロジェクトメンバー（担当者）をデータベースに登録する機能です。

#### フロー

```
1. ユーザーが「担当者登録」等のキーワードを送信
    ↓
2. 担当者登録フォームが表示される
    - 担当者選択（Workspace ディレクトリからドロップダウン）
    - ロール選択
    ↓
3. 「登録する」ボタンを押すと：
    - 担当者マスタに新規行が追記される
    - 担当者IDが自動採番される（M0001, M0002, ...）
    - 確認メッセージが表示される
```

#### 担当者選択

- **通常時**: Google Workspace ディレクトリ（People API）から全ユーザーを取得し、ドロップダウンで選択可能
- **フォールバック**: ディレクトリ取得に失敗した場合は、担当者名とメールアドレスの手動入力に切り替わる

#### ロール一覧

| ロール名 | 説明 |
|---|---|
| コンセプトWG | コンセプトワーキンググループ |
| 原価・配合WG | 原価・配合ワーキンググループ |
| コピー・外装WG | コピー・外装ワーキンググループ |
| 営業 | 営業担当 |
| 代表取締役 | 承認者 |
| その他 | 上記に該当しない場合 |

#### 担当者マスタのデータ構造

| カラム | 説明 |
|---|---|
| 担当者ID | 自動採番（M0001形式） |
| 担当者名 | 氏名 |
| メールアドレス | Workspace メールアドレス |
| ロール | 所属ロール |
| ステータス | 有効 / 無効 |
| 登録日 | 登録日時 |

### 3.4 ワークフロー定義管理

商品開発の作業フロー（タスクのひな形）をスプレッドシートで管理する機能です。スプレッドシートを直接編集することでワークフローを柔軟に変更できます。

#### ワークフロー定義スプレッドシートの構成

| シート名 | 用途 |
|---|---|
| 新商品 | 新商品開発のワークフロー定義 |
| リニューアル・軽微な変更 | リニューアル開発のワークフロー定義 |
| 担当者・ロール定義 | ロールごとの担当者マッピング |
| ディレクトリ（非表示） | People API から取得したユーザー一覧 |

#### ワークフローシートのカラム定義

| カラム | 説明 |
|---|---|
| 親タスク番号 | フェーズ番号（1, 2, 3, ...） |
| 親タスク名 | フェーズ名（例: 基本情報作成、承認1） |
| 子タスク番号 | フェーズ内のタスク番号（空欄の場合は親タスク自体がタスク） |
| 子タスク名 | 個別タスク名（例: 配合作成、原価試算） |
| ロール | 担当ロール |
| 担当者（正） | 正担当者名（担当者・ロール定義シートからドロップダウン選択） |
| 担当者（副） | 副担当者名（担当者・ロール定義シートからドロップダウン選択） |
| 依存タスク | 先行タスクのID（カンマ区切り。例: `2`, `3-1,3-2`） |
| 開始時の挙動 | タスク開始時に実行するアクション |
| 完了時の挙動 | タスク完了時に実行するアクション（通知など） |

**担当者の選択について:**
- 「担当者（正）」「担当者（副）」列は、「担当者・ロール定義」シートに登録された担当者名のドロップダウンから選択できます
- 担当者・ロール定義シートで担当者を追加・変更すると、ワークフローシートのドロップダウンにも自動的に反映されます

**依存タスクの書き方:**
- タスクIDは「親タスク番号」（例: `2`）または「親タスク番号-子タスク番号」（例: `3-1`）形式
- 複数の依存がある場合はカンマで区切る（例: `3-1,3-2,3-3`）
- このタスクを開始するには、指定した依存タスクがすべて完了している必要があります

#### ワークフロー例：新商品

```
1. 基本情報作成
   └─ 基本情報入力 [コンセプトWG]

2. 承認1 [代表取締役]  ← 依存: 1

3. 試算確定（すべて依存: 2）
   ├─ 1. 配合作成 [原価・配合WG]
   ├─ 2. 原価試算 [原価・配合WG]
   ├─ 3. 売価設定 [営業]
   ├─ 4. デザイン案作成 [コピー・外装WG]
   └─ 5. 品質基準策定 [品質管理]

4. 承認2 [代表取締役]  ← 依存: 3-1,3-2,3-3,3-4,3-5

5. 製造準備（すべて依存: 4）
   ├─ 1. 製造指示書作成 [原価・配合WG]
   ├─ 2. パッケージ入稿 [コピー・外装WG]
   └─ 3. 販促物作成 [営業]

6. 最終承認 [代表取締役]  ← 依存: 5-1,5-2,5-3
```

#### ワークフロー例：リニューアル・軽微な変更

```
1. 変更内容整理
   ├─ 1. 変更箇所の特定 [コンセプトWG]
   └─ 2. 影響範囲の確認 [コンセプトWG]

2. 承認 [代表取締役]  ← 依存: 1-1,1-2

3. 変更実施（すべて依存: 2）
   ├─ 1. 配合・原価修正 [原価・配合WG]
   └─ 2. デザイン修正 [コピー・外装WG]

4. 最終確認 [代表取締役]  ← 依存: 3-1,3-2
```

#### 担当者・ロール定義シート

ワークフロー上のロールと実際の担当者を紐付けるシートです。

| カラム | 説明 |
|---|---|
| ロール | ロール名（コンセプトWG、営業 等） |
| 担当者名 | People API のドロップダウンから選択 |
| メールアドレス | 担当者名を元に VLOOKUP で自動入力 |
| 正/副 | 正担当か副担当かの区分 |

**便利な機能:**
- 「担当者名」列は Google Workspace ディレクトリから取得したユーザー一覧のドロップダウンになっています
- 担当者名を選択すると、メールアドレスが VLOOKUP で自動的に入力されます

## 6. 管理者向け操作ガイド

### 6.1 GAS エディタから実行する関数一覧

| 関数名 | 実行タイミング | 説明 |
|---|---|---|
| `authorizeScopes()` | 初回セットアップ時 / スコープ変更後 | OAuth 認可を実行 |
| `setupDatabase()` | 初回セットアップ時 | DB 用シートを初期化 |
| `createWorkflowSheet()` | 初回セットアップ時 | ワークフロー定義スプレッドシートを作成 |
| `syncDirectory()` | メンバー変更時 / 定期的 | ディレクトリ（ユーザー一覧）を最新に更新 |

### 6.2 ワークフローの変更方法

1. ワークフロー定義スプレッドシートを開く
2. 対象のシート（「新商品」「リニューアル・軽微な変更」）を選択
3. タスク行を追加・編集・削除
4. 保存（自動保存）

**注意事項:**
- 親タスク番号は昇順で入力してください
- 同じ親タスク番号を持つ行が複数ある場合、子タスクとして扱われます
- 子タスク番号が空欄の行は、親タスク自体がタスクとなります（承認フェーズなど）
- 「依存タスク」列にはカンマ区切りでタスクIDを入力してください（例: `2`, `3-1,3-2`）
- タスクIDは「親タスク番号」または「親タスク番号-子タスク番号」形式です
- 「担当者（正）」「担当者（副）」列は担当者・ロール定義シートの担当者からドロップダウンで選択できます

### 6.3 担当者・ロールの変更方法

1. ワークフロー定義スプレッドシートの「担当者・ロール定義」シートを開く
2. 「担当者名」列のドロップダウンから担当者を選択
3. メールアドレスが自動入力されることを確認
4. 必要に応じて行を追加（新しいロールの追加など）

### 6.4 ディレクトリの同期

新しいメンバーが入社した場合や、ディレクトリ情報を更新したい場合：

1. GAS エディタで `syncDirectory` 関数を実行
2. ディレクトリ（非表示）シートが最新のユーザー一覧で更新される
3. 担当者名列のドロップダウンも自動的に更新される

**ヒント:** GAS のトリガー機能を使って `syncDirectory` を定期実行（例: 毎日）に設定することもできます。

### 6.5 デプロイ手順

コード変更後の再デプロイ手順：

```bash
# 1. GAS にプッシュ
clasp push --force

# 2. デプロイを更新
clasp deploy -i <デプロイメントID>
```

デプロイ後、Chat Bot が新しいコードで動作するようになります。

## 9. トラブルシューティング

### 9.1 「server error」が表示される

**原因:** GAS の組み込みサービス（`SpreadsheetApp`, `DriveApp`）は、Workspace Add-on の Chat イベントハンドラ内では動作しません。

**対応:** すべての Google API 呼び出しは REST API（`UrlFetchApp.fetch()`）経由で行う必要があります。本システムではすべて対応済みです。

### 9.2 担当者選択ドロップダウンが空になる

**原因:** People API へのアクセス権限がないか、`directory.readonly` スコープが認可されていません。

**対応:**
1. `appsscript.json` に `directory.readonly` スコープが含まれていることを確認
2. GAS エディタで `authorizeScopes` を再実行
3. 認可画面ですべてのアクセスを承認

### 9.3 ワークフロー読み取りエラー

**原因:** `WORKFLOW_SPREADSHEET_ID` が未設定、またはスプレッドシートが削除されています。

**対応:**
1. GAS エディタで `createWorkflowSheet` を実行してスプレッドシートを再作成
2. スクリプトプロパティに `WORKFLOW_SPREADSHEET_ID` が自動設定されることを確認

### 9.4 ファイルがマイドライブに作成される

**原因:** Drive API の `parents` パラメータが正しく設定されていないか、フォルダへのアクセス権がありません。

**対応:**
1. `SpreadsheetDB.js` の `PROJECT_FOLDER_ID` が正しいフォルダ ID であることを確認
2. 実行ユーザーがそのフォルダに書き込み権限を持っていることを確認

### 9.5 Sheets API エラー (403)

**原因:** GCP プロジェクトで Google Sheets API が有効になっていません。

**対応:** GCP コンソールで Google Sheets API を有効にしてください。

## 10. 技術仕様

### 10.1 API 呼び出しパターン

本システムでは、すべての Google API 呼び出しに以下のパターンを使用しています：

```javascript
var token = ScriptApp.getOAuthToken();
var response = UrlFetchApp.fetch(url, {
  method: 'post',       // or 'get'
  headers: { 'Authorization': 'Bearer ' + token },
  contentType: 'application/json',
  payload: JSON.stringify(data),
  muteHttpExceptions: true
});
```

### 10.2 Chat レスポンス形式

Workspace Add-on として動作するため、すべてのレスポンスは以下の形式でラップされます：

```javascript
// 新規メッセージ作成
{
  hostAppDataAction: {
    chatDataAction: {
      createMessageAction: {
        message: { text: '...' }  // or { cardsV2: [...] }
      }
    }
  }
}

// 既存メッセージ更新
{
  hostAppDataAction: {
    chatDataAction: {
      updateMessageAction: {
        message: { cardsV2: [...] }
      }
    }
  }
}
```

### 10.3 Card V2 UI

Google Chat のカード UI は Card V2 形式で定義されています。主に以下のウィジェットを使用：

- `textInput` - テキスト入力フィールド
- `selectionInput` (DROPDOWN) - ドロップダウン選択
- `dateTimePicker` (DATE_ONLY) - 日付選択
- `buttonList` - アクションボタン

### 10.4 イベントルーティング

```
onMessage(event)
  ├─ 「担当者登録」「メンバー登録」 → getMemberRegistrationFormCardsV2_()
  ├─ 「登録」「立ち上げ」「新規」「プロジェクト」 → getProjectTypeSelectionCardsV2_()
  └─ その他 → ヘルプメッセージ

onCardClick(event)
  ├─ onProjectTypeSelected → 種別に応じたフォーム表示
  ├─ onProjectInfoSubmitted → プロジェクト登録処理
  └─ onMemberRegistrationSubmitted → 担当者登録処理
```
